JVM 
- implementation of a vm which executes a Java program
- JVM (interprets) -------> bytecode
- stores class info in memory area.
- executes bytecode generated by java compiler.
- manipulates various memory areas at runtime

- COMPONENTS
	- Class Loaders
		- loads, verifies and links bytecode
	- Runtime Data Areas (defined by JVM)
		- memory areas to execute a Java program
		- used during runtime
		- some created on JVM startup and destroyed when JVM exits
		- some created and destroyed with thread
		- Method Area
			- stores runtime constant pools, field, method data
			- code for methods and constructors
			- fully qualified class names
			- created when JVM starts
			- all JVM threads share this mem area
		- Heap Area
			- memory allocated by JVM for all class instances and arrays
			- GC reclaims heap memory for objects
			- Eden Space
				- part of Young Generation space
				- JVM allocates memory from when object is created
			- Survivor Space
				- part of Young Generation space
				- contains existing objects that survived minor GC
			- Tenured Space
				- Old Generation space
				- holds long surviving objects
				- moved here when young generation objects meet threshold
		- Stack Area
			- stores data as frames and each frame stores local variables, partial results and nested method calls
			- created by JVM whenever new thread is created
			- private for each thread
			- entries in the stack are called stack frames
			- frame contains 3 parts
				- Local Variable Array
					- contains all local variables and parameters of the method
				- Operand Stack
					- workspace for storing intermediate calculations result
				- Frame Data
					- stores partial results
					- stores return values for methods
					- stores reference to the Exception table
		- PC Registers
			- each JVM thread has a separate PC Register
			- stores the address of currently executing instruction
		- Native method stacks
			- written in languages other than java
			- also knows as C stacks
			- store native method information
			- used to keep track of the state of native methods whenever they compile into machine codes
			- created by JVM to create a new thread
	- Execution Engine
		- Interpreter
			- executes the instructions using information present in memory areas
			- executes the loaded and verified bytecode
			- when one method is called multiple time, new interpretation is required every time (uses JIT compiler to mitigate this)
		- Just-in-Time (JIT) Compiler
			 - compiles bytecode of the often-called methods into native code at run-time
			 - responsible for the optimization of Java programs
		- Garbage Collector
			- used for memory management
			- deleting unused objects by looking at heap memory
			- is a daemon thread
			- can be called explicitly
				- using Systemgc() method
			- decided by JVM on when to invoke
	- Java Native Interface
		- acts as an interface between Java code and the native C/C++ libraries
		- enables native methods to call code running in the JVM
	- Native Libraries
		- platform specific libraries
		- contains implementation of native methods


JRE
- bindle of software components used to run Java applications
- Core Components
	- an implementation of JVM

	- property files
- Bootstrap Classes
	- found under jre/lib/
	- includes

		- net.properties

		- jre/lib/fonts

JDK (Java Development Kit)
- provides environment and tools for developing, compiling, debugging and executing a java program
- Components
	- JRE
	- Development Tools
		- Basic Tools
			- utilities for compiling, debugging, archiving, generating Javadocs, etc
			- javac
				- reads class and interface definitions and compiles them into class files
			- java
				- launches java application
			- javadoc
				- generates HTML pages of API documentation from Java source files
			- apl
				- finds and executes annotation processors based on the annotations present in the set of specified source files
			- appletviewer
				- enables to run Java applets without a web browser
			- jar
				- packages java applets/applications into a single archive
			- jdb
				- used to find and fix bugs in Java applications
			- javah
				- produces C header and source files from a Java class
			- javap
				- disassembles and displays info about fields, constructors and methods in a class file
			- extcheck
				- detects version conflicts btw target Java Archive(JAR) file and currently installed extension JAR files
		- Security Tools
			- include key and certificate management tools, used to manipulate Java Keystrokes
			- keytool
				- helps in managing keystore entries, cryptographic keys and certificates
			- jarsigner
				- generates digitally signed JAR files by using keystroke info
			- policytool
				- enables us to manage the external policy config files

				- kinit
					- used to obtain and cache Kerberos TGTs
				- ktab
					- manages principle names and key pairs in key table
				- klist
					- displays entries in the local credentials cache and key table
		- Internationalization Tool

		- Remote Method Invocation (RMI) Tools
			- enable remote communication between Java applications thus providing scope for development of distributed applications

				- generates stub, skeleton and tie classes for remote objects using Java Remote Method Protocol (JRMP) or Inter-Orb Protocol (IIOP)
			- rmiregistry
				- creates and starts remote object registry
			- rmid
				- starts activation system daemon
				- allows objects to be registered and activated in a Java Virtual Machine
			- serialver
				- returns serial version UID for specified classes

**OOP Pillars**
	- **Encapsulation**
		- Bundle data (fields) and methods. Hide fields with `private`, expose via `public` getters/setters.
		- _Example:_ `Employee` class with private `salary`, public `getSalary()`
	- **Inheritance**
		- `class Dog extends Animal`. Use `super` to call parent constructor/methods.
		- _Prefer composition over inheritance_: "Has-a" vs "Is-a". Composition (e.g., `Car` has an `Engine`) is more flexible.
	- **Polymorphism**
		- **Compile-time (overloading)**: same method name, different parameters
		- **Runtime (overriding)**: subclass provides specific implementation.
			- _Interview Q:_ "Can you override a private method?" No – private methods are not inherited.  
			- "Static methods?" No – they are hidden, not overridden.
	- **Abstraction**
		- **Abstract class**: can have state, constructors, and concrete methods. Use when classes share code.
		- **Interface** (Java 8+): can have default/static methods. Use for capabilities (e.g., `Runnable`).
			- _Example:_ `Animal` abstract class with `abstract void sound()`; `Pet` interface with `default void play()`.
- ### Collections & Generics
	- **Key Interfaces**: `List`, `Set`, `Map`.
	- **ArrayList**
		- resizable array, fast random access
	- **LinkedList**
		- doubly-linked, fast insertion/deletion
	- **HashSet**
		- uses `hashCode()` and `equals()` for uniqueness, no order.
	- **HashMap**
		- key-value pairs, allows one null key

**Generics**: Type safety.  
`List<String> list = new ArrayList<>();`

- **Stack**:
    
    - Primitive data types (`int`, `boolean`, `double`)
        
    - Object **references** (pointers to objects in heap)
        
    - One stack frame per method call. LIFO.
        
    - _Thread private_.
        
    - _Error_: `StackOverflowError` (infinite recursion).

**Heap**:

- All **actual objects** (instances of classes, arrays).
    
- Shared across all threads.
    
- Managed by Garbage Collector (GC).
    
- _Error_: `OutOfMemoryError`

- **Method Area**:
    
    - Class metadata, static variables, constants.


- **HashMap internal**: array of buckets, `hashCode()` determines bucket, `equals()` compares keys.
    
- **HashSet** uses HashMap internally.
    
- **ConcurrentModificationException**: when modifying collection while iterating.

**Exception Hierarchy**:

- `Throwable` → `Exception` (checked) → `RuntimeException` (unchecked) → `Error`.
    
- Checked: must be caught or declared (e.g., `IOException`).
    
- Unchecked: programming errors (e.g., `NullPointerException`).
    
- **Try-with-resources**: automatically closes resources (e.g., `BufferedReader`

HASHMAP
import java.util.HashMap;  
import java.util.Map;  
  
public class map {  
    public static void main(String[] args){  
        String sentence = "Java is powerful and Java is popular and Java is object oriented";  
  
        sentence = sentence.toLowerCase();  
  
        String[] words = sentence.split(" ");  
  
        HashMap<String, Integer> wc = new HashMap<>();  
  
        for (String word : words){  
            if (wc.containsKey(word)){  
                wc.put(word, wc.get(word) + 1);  
            }  
            else {  
                wc.put(word, 1);  
            }  
        }  
        System.out.println("Word Frequencies");  
        for (Map.Entry<String, Integer> entry : wc.entrySet()){  
            System.out.println(entry.getKey() + " : " + entry.getValue());  
        }  
  
    }  
}


**Memory (Stack vs Heap)**

- **Stack**: stores primitives and object references; each method call pushes a frame; local variables.
    
- **Heap**: stores all objects; shared across threads.
    
- **Garbage Collection**: objects become eligible when no references point to them.
    

**Interview Q**: "What causes StackOverflowError?" (infinite recursion). "OutOfMemoryError?" (heap full).

### equals() & hashCode() Contract

**Contract**:

- If `equals()` returns true, `hashCode()` must return the same value.
    
- If `hashCode()` same, objects may or may not be equal.
    
- Override both for correct behavior in collections (HashMap, HashSet).
    

**Why important**: Without proper override, duplicates can appear in HashSet.
